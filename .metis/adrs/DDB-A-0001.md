# ADR-1: SQLite timestamp representation: RFC3339 TEXT

## Context

`types::Timestamp` wraps `DateTime<Utc>`. Postgres stores it natively as `timestamptz`. SQLite has no native timestamp type — it must be encoded into one of TEXT / INTEGER / REAL. The encoding is a **wire-format decision that is effectively permanent**: once rows exist, changing it is a data migration. It also determines whether range queries and `ORDER BY` behave correctly, because SQLite compares the stored representation directly.

The hard requirement: **lexicographic (byte/string) ordering of the stored value must equal chronological ordering**, so `WHERE ts > $x` and `ORDER BY ts` work without special handling on the SQLite arm. This blocks the Timestamp type task ([[DDB-T-0008]]).

## Decision

Store `types::Timestamp` on SQLite as **RFC 3339 / ISO 8601 TEXT**, with a **fixed, normalized format**:

- **UTC only** — convert to UTC before formatting (the wrapper is `DateTime<Utc>`, so this is total).
- **`Z` suffix** (literal), never a numeric offset — a fixed-width zone marker keeps all values mutually comparable.
- **Fixed fractional-second precision** — always render the same number of subsecond digits (e.g. microseconds, 6 digits: `%Y-%m-%dT%H:%M:%S%.6fZ`), zero-padded. Variable precision breaks lexicographic ordering (`...:01Z` sorts after `...:01.5Z`).
- Fixed-width fields throughout (zero-padded year/month/day/h/m/s), so every stored string is the same length and string comparison == time comparison.

Postgres continues to use native `timestamptz` (delegate to diesel's chrono support).

## Alternatives Analysis

| Option | Pros | Cons | Risk | Cost |
|--------|------|------|------|------|
| **RFC3339 TEXT, fixed format (chosen)** | Human-readable/debuggable; sorts correctly if format is fixed; trivial chrono format/parse | ~27 bytes/value; must rigidly enforce one format | Low (if format frozen) | Low |
| Unix INTEGER (micros since epoch) | Compact (8 bytes); sorts correctly inherently; fast range queries | Opaque on inspection; precision/epoch choice baked in; negative pre-1970 values | Low | Low |
| REAL (Julian day, SQLite default for some funcs) | Native to some SQLite date funcs | Float precision loss; surprising rounding | Medium | Low |

## Rationale

RFC3339 TEXT wins on **debuggability** — values are legible in `sqlite3` shells, dumps, and logs, which matters for a library others adopt and inspect. Its one real weakness (correct sorting depends on a fixed format) is fully neutralized by freezing the format here and enforcing it in one `ToSql` impl. The compactness/perf edge of unix INTEGER is real but not worth the opacity for v1; the ADR records it as the documented fallback if a workload proves range-query/index performance demands it.

## Consequences

### Positive
- Sortable, range-queryable timestamps on SQLite with no per-query special-casing.
- Legible stored values; easy debugging and manual data inspection.
- Symmetric with the human-readable spirit of the crate.

### Negative
- ~27 bytes per value vs 8 for INTEGER (index size + IO); acceptable for v1.
- The fixed format is load-bearing: any future code path that writes a differently-formatted string silently corrupts ordering. Mitigation: a single canonical `ToSql` formatter + round-trip and ordering tests in [[DDB-T-0008]].
- Sub-second precision is capped at the chosen digit count; values with finer precision are truncated (documented).

### Neutral
- Postgres side is unaffected (native `timestamptz`).
- SQLite 3.45+ has date/time helpers, but we store/compare the raw TEXT ourselves, so we don't depend on them.

## Review Schedule

### Review Triggers
- A real workload shows SQLite timestamp range queries / index size are a bottleneck → reconsider unix INTEGER.
- A need for sub-microsecond precision beyond the frozen digit count arises.

Otherwise permanent (changing it is a data migration).
