# ADR-1: In-crate interop limits: native-type shim & unified ON CONFLICT

## Context

Two often-requested capabilities can't be made to work cleanly **inside this crate**, because both run into limits of diesel's `#[derive(MultiConnection)]` / the orphan rule. We investigated both (2026-06-09/10) and need a recorded decision so they aren't re-litigated.

**1. Native-type shim** â€” let users type columns with diesel's *native* sql types (`diesel::sql_types::Uuid`/`Jsonb`/`Timestamptz`) and have them work on both backends with no portable markers.
- Blocked by the **orphan rule**: `impl ToSql<diesel::sql_types::Uuid, Sqlite> for uuid::Uuid` is foreign-trait + foreign-types â†’ `error[E0117]` (verified with a compile probe).
- Diesel also provides **no** SQLite support for these PG-native types â€” there's nothing to "enable." A fork wouldn't add it (diesel intentionally scopes them to Postgres).

**2. Unified `ON CONFLICT` (upsert) through `MultiBackend`** â€” `on_conflict(...)` doesn't compile through `DualConnection` (verified, see [[DDB-T-0021]]).
- `#[derive(MultiConnection)]` hard-sets `MultiBackend`'s `SqlDialect::OnConflictClause = DoesNotSupportOnConflictClause`. There's no Cargo feature to flip it (unlike `RETURNING`, fixed by `returning_clauses_for_sqlite_3_35`), and the dialect is an associated type on generated code we can't re-impl.

## Decision

**Ship neither in-crate.** Keep the portable types as the answer for type interop, and the escape hatch (`DualConnection::dispatch` / `match conn`) as the answer for upsert. Pursue the only real fix for upsert â€” enabling on-conflict on `MultiConnection`'s `MultiBackend` â€” as an **upstream diesel contribution** (issue/PR drafted below). The native-type shim is **won't-fix** (no viable path, in-crate or upstream).

## Alternatives Analysis

| Option | Pros | Cons | Risk | Cost |
|--------|------|------|------|------|
| **Keep dispatch + upstream (chosen)** | Already shipped; no fork/maintenance; fixes upstream for everyone | Upsert needs a 2-line `dispatch` block; depends on diesel accepting the PR | Low | ~0 (+ a PR) |
| Local upsert builder (`QueryFragment<MultiBackend>` for a local wrapper) | Publishable; native-feeling `dualdb` upsert; no fork | Reimplements diesel's upsert fragment (binds, conflict targets, `excluded.`); easy to miscompile | Medium-High | Few hundred LOC |
| Hand-roll `MultiBackend` (drop the derive, own `SqlDialect`) | Most powerful; flips on-conflict to `PgLike` "natively" | Vendors ~1â€“2k lines of backend internals; must track every diesel release | High | Very high, ongoing |
| `[patch.crates-io]` a forked diesel | Trivial for our repo | **Doesn't reach published consumers** â€” `[patch]` isn't part of our published manifest; downstream gets stock diesel | â€” | Dev-only, not a real fix |

## Rationale

The escape hatch already makes upsert a clean two-liner, so the marginal value of a "native" in-crate upsert is small â€” and the two in-crate routes to it are either fragile (local builder) or a permanent maintenance tax (hand-rolled `MultiBackend`). The `[patch]` shortcut is a non-fix because patches don't propagate to people who depend on us from crates.io. The proper fix is one associated type in diesel's derive; contributing it upstream helps the whole ecosystem and keeps our crate thin. The native shim has no path at all (orphan rule + diesel's deliberate scoping), so the portable types stand as the answer.

## Consequences

### Positive
- Crate stays thin: just `#[derive(MultiConnection)]` + our bridge, no vendored internals to maintain.
- Honest, documented boundaries; upsert has a working, visible idiom today.

### Negative
- Upsert isn't "write-once" â€” it needs an explicit per-backend `dispatch`/`match` (loud divergence, by design).
- A real unified upsert waits on an upstream diesel change we don't control.

### Neutral
- If diesel adds the on-conflict dialect to `MultiConnection` (or a feature like the RETURNING one), upsert through `DualConnection` would "just start working" with no API change on our side â€” revisit then.

## Upstream issue/PR (draft â€” to file when the repo has a remote/maintainer bandwidth)

> **Title:** `MultiConnection`: enable `ON CONFLICT` on the generated `MultiBackend`
>
> **Problem:** `#[derive(MultiConnection)]` over `PgConnection` + `SqliteConnection` produces a `MultiBackend` whose `SqlDialect::OnConflictClause` is `DoesNotSupportOnConflictClause`, so `insert_into(t).on_conflict(...).do_update()/.do_nothing()` fails to compile through the multi-connection â€” even though **both** backends support `ON CONFLICT (col) DO UPDATE/NOTHING` with the same `PgLike` syntax (SQLite â‰Ą 3.24). Compare `returning_clauses_for_sqlite_3_35`, which already lets RETURNING work through `MultiBackend`.
>
> **Proposal:** when every connection in the derive has a `PgLike` on-conflict dialect, set the generated `MultiBackend`'s `SqlDialect::OnConflictClause = PgLikeOnConflictClause` (ideally gated like the RETURNING feature). The `MultiBackend` query builder emits the same SQL string to whichever backend is active, which both accept.
>
> **Repro:** `#[derive(MultiConnection)] enum C { Pg(PgConnection), Sqlite(SqliteConnection) }`, then `insert_into(t).values(..).on_conflict(t::id).do_nothing().execute(&mut c)` â†’ `DoesNotSupportOnConflictClause: PgLikeOnConflictClause` is not satisfied.
