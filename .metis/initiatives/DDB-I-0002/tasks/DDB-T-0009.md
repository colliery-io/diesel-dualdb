# Bytes portable type (concrete + bridge + tests)

## Parent Initiative

[[DDB-I-0002]]

## Objective

Add `types::Bytes(Vec<u8>)` as a full vertical slice (marker, concrete `Pg`/`Sqlite`, bridge, tests). Pg `bytea`, SQLite `BLOB`. The simplest remaining type — no dependency on optional crates (uses `std`), so it also exercises the bridge with no extra feature gate.

## Acceptance Criteria

## Acceptance Criteria

- [x] `sql_types::Bytes` marker (`bytea` / `Binary`), no feature gate.
- [x] `types::Bytes(pub Vec<u8>)` (`src/types/bytes.rs`) + `From` conversions; ungated.
- [x] Concrete impls delegate to diesel's `Binary`/`Vec<u8>` on both backends.
- [x] Bridge: three one-liner impls on `MultiBackend` (ungated `mod bytes` in `src/backend.rs`).
- [x] Round-trip tests both backends: empty, mixed bytes (NUL + 0xFF + 0x80), 1 MiB.
- [x] **Empty-vs-NULL** tested: `Some(Bytes(vec![]))` reads back as `Some(empty)`, `None` as `None` — distinct.
- [x] One-arm ORM test through `DualConnection`, both backends — the macro runs all four ways (concrete sqlite/pg + bridge sqlite/pg).

## Implementation Notes

### Technical Approach
Both sides delegate to diesel's existing `Binary` support, so this is the lightest type. The interesting bit is the empty-vs-NULL semantics (SQLite is historically loose here) — assert it. Reuse spike test patterns.

### Dependencies
[[DDB-T-0006]] (no feature gate needed, but follows the module convention). Bridge pattern from [[DDB-T-0003]]. Independent of the other type tasks.

### Risk Considerations
Low. Watch SQLite empty-blob vs NULL coercion; the dedicated test covers it.
