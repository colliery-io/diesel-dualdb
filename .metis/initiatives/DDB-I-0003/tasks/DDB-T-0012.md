# #[dualdb::test(pg, sqlite)] attribute macro + migrate one test file

## Parent Initiative

[[DDB-I-0003]]

## Objective

The headline ergonomic: `#[dualdb::test(pg, sqlite)]` expands one test body into a per-backend `#[test]`, each handed a `&mut DualConnection`, so both-backend tests are a single annotation instead of the hand-rolled 4-way macro the type tests currently repeat. Highest payoff of M1; built first.

## Acceptance Criteria

- [x] `#[proc_macro_attribute] pub fn test` in `diesel-dualdb-macros`, re-exported as `diesel_dualdb::test`; used as `#[diesel_dualdb::test(pg, sqlite)]`.
- [x] On `fn name(conn: &mut DualConnection) { ‚Ä¶ }` generates `name_sqlite` (in-memory) and `name_pg` (env-gated; skip notice + return when `DUALDB_PG_URL` unset). Verified both: under `angreal test all` pg variants run against the container; under `angreal test sqlite` they skip (still pass).
- [x] Backend selection: comma list of `pg`/`sqlite`; empty = both; unknown ident ‚Üí a spanned `compile_error!`.
- [x] User attributes (`#[should_panic]`, docs) forwarded to the generated `#[test]` fns; the kept inner fn is stripped of them.
- [x] Migrated `tests/bytes.rs` ‚Äî proof. (See note.)
- [x] `angreal test all` green (20 tests); `angreal check all` clean (incl. `--workspace` clippy).

**Migration note:** the old `tests/bytes.rs` ran a macro 4 ways (concrete `Pg`/`Sqlite` **and** `DualConnection`). `#[dualdb::test]` produces only the `DualConnection` (bridge) path ‚Äî which *dispatches through* the concrete `ToSql`/`FromSql`, so the concrete impls are still exercised, just not in isolation. Consciously dropped the concrete-only variants (allowed by the AC); net test count unchanged (2 fns √ó 2 backends = 4). Per-backend DDL (BLOB vs BYTEA) is selected by a small `match &*conn` in a `create_table` helper ‚Äî the one legitimately divergent bit; queries stay backend-agnostic.

*(trybuild compile-fail check: not added ‚Äî nice-to-have; the unknown-ident path emits `compile_error!` but isn't asserted by a test yet.)*

## Implementation Notes

### Technical Approach
`#[proc_macro_attribute]` parsing `attr` as a comma list of `pg`/`sqlite` idents (default both) and `item` as an `ItemFn` (via `syn`). Emit, per selected backend, a `#[test] fn <name>_<backend>()` that sets up the connection and calls the original body. Two shapes to decide: (a) keep the user fn and call it from generated wrappers, or (b) inline the block into each wrapper. Inlining avoids an unused-fn warning. The connection-setup snippet hard-codes `:memory:` for sqlite and `std::env::var("DUALDB_PG_URL")` for pg ‚Äî mirror the current test files exactly. Consider an optional `setup = path::to_fn` arg later (for DDL) but not required for v1 (tests can create tables in the body).

Naming/path: the macro lives in `diesel-dualdb-macros`; `diesel-dualdb` does `pub use diesel_dualdb_macros::test;` so it's `diesel_dualdb::test`. (The design's `#[dualdb::test]` assumes a `use diesel_dualdb as dualdb;` alias or rename ‚Äî document the real path.)

### Dependencies
[[DDB-T-0011]] (the macro crate must exist). Independent of the bridge-codegen macro.

### Risk Considerations
Medium. `&mut DualConnection` body shape must be parsed/forwarded correctly. The macro hard-codes connection setup ‚Äî keep it identical to the proven test files. Hygiene: generated code references `diesel_dualdb::DualConnection`, `diesel::Connection` ‚Äî emit fully-qualified paths to avoid import assumptions in the user's test file.

**Design choices:**

- Connection setup mirrors the proven test files exactly (`:memory:` / `DUALDB_PG_URL`).
- **Path is `diesel_dualdb::test`, not the design's `dualdb::test`** ‚Äî no rename/alias forced on users; documented in the macro + crate rustdoc.
