# Migrate remaining types + tests onto the macros

## Parent Initiative

[[DDB-I-0003]]

## Objective

Finish the migration: every portable type uses the bridge-codegen macro (no hand-written `mod <t>` blocks left in `src/backend.rs`), and the type test files use `#[dualdb::test]`. Closes M1 — adding a type or a both-backend test is now one annotation.

## Acceptance Criteria

## Acceptance Criteria

- [x] `Bytes` and `Timestamp` bridges migrated to `crate::bridge!(...)`; `Json` stays hand-written (generic — descoped in [[DDB-T-0013]]). `src/backend.rs` is now 3 `bridge!` lines + the one Json block.
- [x] Type test files migrated to `#[diesel_dualdb::test]`: `bytes.rs` (T-0012), `timestamp.rs`, `json.rs`, and `uuid.rs` (consolidates the old `types.rs` round-trip + `bridge.rs` RETURNING). The 4-way macros are retired. `connectivity.rs`/`pool.rs` kept as-is (connection/pool mechanics, not type round-trips).
- [x] Suite stays green — **20 tests** (same count, no coverage regression); `angreal check all` clean across the feature matrix.
- [x] `README.md` with a worked "Adding a portable type" example (marker + newtype + one-line `bridge!`) plus usage and `#[diesel_dualdb::test]` snippets.

## Implementation Notes

### Technical Approach
Mechanical once [[DDB-T-0012]] and [[DDB-T-0013]] land. Migrate one type at a time, running `angreal test all` after each. Json is the canary (generic + bounds) — if the bridge macro can't express it, leave Json hand-written and note the limitation. Keep the connectivity/pool tests as-is if they don't fit `#[dualdb::test]` (they assert connection/pool mechanics, not type round-trips).

### Dependencies
[[DDB-T-0012]] and [[DDB-T-0013]] (both macros must exist). Last task of M1.

### Risk Considerations
Low-medium. Mostly find-and-replace, but the Json generic bridge is the one that may resist the macro. Don't regress coverage — diff test counts before/after.
