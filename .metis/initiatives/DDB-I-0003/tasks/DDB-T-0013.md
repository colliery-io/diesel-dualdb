# Bridge-codegen macro + migrate Uuid

## Parent Initiative

[[DDB-I-0003]]

## Objective

Replace the hand-written `MultiBackend` bridge blocks in `src/backend.rs` with a macro, so adding a portable type no longer means hand-copying three impls. Prove it on `Uuid` first (reference type), confirming the macro reproduces the exact current behavior.

## Acceptance Criteria

## Acceptance Criteria

- [x] Function-like `#[proc_macro] bridge!(Marker, Newtype)` in `diesel-dualdb-macros`, re-exported as `diesel_dualdb::bridge`, generating the three `MultiBackend` impls (`HasSqlType` via `lookup_sql_type`, `ToSql` via `set_value((ST, self))`, `FromSql` via `from_sql::<Self, ST>()`).
- [~] **Generic newtype (Json): consciously descoped** (the task pre-authorized this fallback). Json's split bounds — `ToSql` needs `Serialize + Debug + Send + Sync + 'static`, `FromSql` needs `DeserializeOwned` — don't fit a clean `bridge!(...)` one-liner; a macro DSL to express them per-impl would be longer than the 12-line hand-written block. **Decision: `bridge!` covers all non-generic types; Json stays hand-written as the documented exception.**
- [x] Migrated `Uuid`'s bridge from the hand-written `mod uuid` block to `crate::bridge!(crate::sql_types::Uuid, crate::types::Uuid)`; bridge/types/pool tests still green.
- [x] Feature-gating works: `#[cfg(feature = "uuid")]` on the `bridge!` invocation; `--no-default-features` (gated off) and `--features uuid` both compile.
- [x] `angreal test all` (20) + `angreal check all` green.

**Design:** chose a **function-like macro** over `#[derive]` — the marker lives on a *different* type (`sql_types::Uuid`) than the newtype (`types::Uuid`), so a derive on the newtype would need to re-read diesel's `#[diesel(sql_type=…)]` helper attr; passing both paths explicitly is clearer. Needed `extern crate self as diesel_dualdb;` in `lib.rs` so the macro's `::diesel_dualdb::MultiBackend` paths resolve internally (and for external users).

## Implementation Notes

### Technical Approach
Expansion target is the existing `mod uuid { … }` block verbatim. A `#[derive(DualBridge)]` is most ergonomic but must extract the marker type and any generic params/bounds from the input. The generic + bounds case (Json) is the hard part; if a derive can't express the `Send+Sync+'static` MultiBackend-only bound cleanly, fall back to a function-like `bridge!` taking explicit bounds. Decide the surface here, on Uuid (no generics), then validate the generic path is expressible before committing.

### Dependencies
[[DDB-T-0011]] (macro crate). Pairs with [[DDB-T-0014]] (migrate the rest). Optional per the initiative — if it proves more complex than the hand-written blocks are worth, this task can be dropped and M1 ships only `#[dualdb::test]`.

### Risk Considerations
Medium-high — generic newtype + feature-gate + MultiBackend-only bound is the fiddly combination. Keep the hand-written blocks until the macro reproduces them exactly. This is the task most likely to be descoped.
