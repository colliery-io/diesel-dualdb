## Status Updates

### 2026-06-10 — Complete

`#[derive(DualEnum)]` works on both backends — built it directly (no separate design pass; the user chose "full + generic"). The derive emits a public `<Enum>SqlType` marker (concrete, so unlike Array no generic-sql-type gymnastics) plus all impls in a `const _: () = { … }` block: Pg ToSql writes the label bytes / FromSql matches `str::from_utf8(value.as_bytes())`; SQLite stores/reads the label as TEXT; the `MultiBackend` bridge mirrors `bridge!` (`set_value((Marker, self))`); hand-written `AsExpression` ×4 + `Queryable` (Row=Self). Attribute parsing via `parse_nested_meta` (`pg_type`, per-variant `rename`).

Gotchas handled: enum must derive `Debug` (`ToSql` supertrait — documented); PG `CREATE TYPE` persists in the test DB so the test does `DROP TYPE IF EXISTS … CASCADE` first. Verified insert + select + `load::<Vec<Mood>>` incl. the renamed `meh` variant, both arms.

**DDB-I-0006 complete** — Decimal + Array + Enum all shipped. The extended type set is in v1.
