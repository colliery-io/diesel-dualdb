## Status Updates

### 2026-06-10 â€” Complete

Fully-generic `Array<T>` works on both backends. The interesting bits (studied diesel 2.3.3 source to get right):
- **Marker must be constructible** â€” diesel's `Array<ST>(ST)` holds an ST value; ours holds `PhantomData<ST>` so `Array::<ST>::default()` can be built for the bind-collector tuple `(marker, &value)`. Hand-wrote `Default` (the derive over-bounds `ST: Default`).
- **Newtype required** â€” impl'ing `AsExpression`/`Queryable` for bare `Vec<T>` conflicts with diesel's blanket `impl AsExpression for T: Expression` (E0119) and the `Nullable` variant hits the orphan rule (E0117). A local newtype `types::Array<T>` sidesteps both.
- **Generic wiring** â€” can't `#[derive(AsExpression, FromSqlRow)]` (sql_type is generic), so hand-wrote `AsExpression` Ă—4 (`diesel::internal::derives::as_expression::Bound`) + `Queryable` (Row=Self via the `FromStaticSqlRow` blanket). `ToSql` has a `Debug` supertrait â†’ element `T: Debug`.
- PG delegates to diesel's `Vec<T>` array impls; SQLite uses `serde_json`. MultiBackend bridge mirrors `Json`'s split Send+Sync bounds.

Both arms green (`Array<String>`, `Array<i32>`, empty). schema-gen array support deliberately deferred (doesn't fit the single-token rewrite). Next: [[DDB-T-0026]] Enum (needs a derive macro + likely a design note first).
