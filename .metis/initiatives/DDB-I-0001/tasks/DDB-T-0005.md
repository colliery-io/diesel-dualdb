# Test orchestration: docker-compose Postgres + angreal

## Parent Initiative

[[DDB-I-0001]]

## Objective

Give the suite a reproducible, disposable Postgres to run against, orchestrated by angreal — "bring your own Postgres container" instead of depending on a host install or a hand-managed `DUALDB_PG_URL`. Unblocks the Pg-side round-trips in [[DDB-T-0002]] and is the concrete foundation for M0's CI story ([[DDB-I-0002]]) and the eventual `#[dualdb::test]` harness ([[DDB-I-0004]]). Added mid-spike at the user's direction.

## Acceptance Criteria

## Acceptance Criteria

- [x] `docker-compose.yml` defines a `postgres:16` service, ephemeral (tmpfs data), published on host port **55432** to avoid clashing with a host Postgres, with a `pg_isready` healthcheck.
- [x] `.angreal/task_test.py` provides: `angreal test all` (up → wait-ready → `cargo test` with `DUALDB_PG_URL` → down), `angreal test sqlite` (no container; Pg tests self-skip), and `angreal db up` / `angreal db down` lifecycle.
- [x] `tests/connectivity.rs` smoke test: SQLite in-memory always; Postgres only when `DUALDB_PG_URL` is set, else self-skip.
- [x] `angreal test sqlite` is green with no container running.
- [x] `angreal test all` brings the container up, runs the suite against real Postgres, and tears it down.
- [x] Verified the Pg test genuinely connects (ran with `--nocapture` + URL set; no skip notice) — not a false green.

## Implementation Notes

### Technical Approach
angreal `DockerCompose` integration drives lifecycle; readiness is polled via `pg_isready` inside the container (`dc.exec`) because `up -d` returns before the DB accepts connections. `DUALDB_PG_URL = postgres://dualdb:dualdb@localhost:55432/dualdb_test`. `cargo test` is invoked as a subprocess with the URL injected into the env.

**Gotcha:** in angreal 2.8.8 `angreal.get_root()` returns a `str`, not a `Path` — wrap with `pathlib.Path(...)` before `.parent` (the skill docs show `.parent` directly, which raises `AttributeError`).

### Dependencies
[[DDB-T-0001]] (crate must build/test). Requires a running Docker daemon for the `all`/`db` tasks; `test sqlite` needs nothing.

### Risk Considerations
Low. If the Docker daemon is down, the `all`/`db` tasks fail with a clear message and `test sqlite` still works. CI will run the same `angreal test all`; bundled-vs-system Postgres client libs noted in [[DDB-T-0001]] are the one thing to watch when this moves to CI.
