# MultiBackend bridge for Uuid + one-arm RETURNING test (the bet)

## Parent Initiative

[[DDB-I-0001]]

## Objective

**The make-or-break step of the whole crate.** Implement the three bridge traits for `types::Uuid` against the generated `MultiBackend`, and prove that `get_result` / insert-with-`RETURNING` actually compile and run through `DualConnection` on one arm, on both backends. If this works, write-once is real and #4079 is closed for our types. If it can't be made to work, the architecture changes.

## Acceptance Criteria

## Acceptance Criteria

- [x] `impl HasSqlType<sql_types::Uuid> for MultiBackend` — delegates to the derive's `Self::lookup_sql_type::<sql_types::Uuid>(lookup)`.
- [x] `impl ToSql<sql_types::Uuid, MultiBackend> for types::Uuid` — `out.set_value((sql_types::Uuid, self))`, dispatched to the active arm.
- [x] `impl FromSql<sql_types::Uuid, MultiBackend> for types::Uuid` — `bytes.from_sql::<Self, sql_types::Uuid>()` via the derive's public `MultiRawValue::from_sql` helper. **Not high-risk after all — the derive pre-solves the RawValue match.**
- [x] `QueryMetadata<sql_types::Uuid>` satisfied for `MultiBackend` (follows from `HasSqlType`).
- [x] **One-arm test, mixed row** (`tests/bridge.rs`): `Item { id: Uuid, name, count }`; `insert_into(...).values(&item).get_result()` (RETURNING) and `table.filter(id.eq(uuid)).get_result()`, through `&mut DualConnection`, on both SQLite (in-memory) and Pg (env-gated, ran for real).
- [x] No per-backend `match` in the test body — `exercise_one_arm(&mut DualConnection, …)` is backend-agnostic.

## Implementation Notes

### Technical Approach
Follow the `MultiConnection` derive docs' `MyEnum`/`MyInteger` extension example precisely — that is the canonical recipe for this. The `FromSql<_, MultiBackend>` impl is where the generated `RawValue` enum must be matched; expand the derive (`cargo expand`) if the variant shape isn't obvious. Mixed row matters: metadata-lookup interactions between portable and native columns hide there, and that's the realistic query shape.

### Dependencies
[[DDB-T-0002]] (concrete Uuid impls) and [[DDB-T-0001]].

### Risk Considerations
**Highest-risk task in v1.** If `FromSql<_, MultiBackend>` can't be expressed (e.g. the generated `RawValue` isn't matchable from outside the derive), capture exactly why in the spike report ([[DDB-T-0004]]) — that's a primary deliverable, success or failure. Consider `cargo expand` early. Fallback options to note if blocked: upstream patch to Diesel, or a different connection-abstraction strategy.

## Status Updates

### 2026-06-08 — Complete — ✅ THE BET PAID OFF (GO)

Write-once `get_result`/`RETURNING` runs through `DualConnection` on one arm, both backends, no per-backend `match`. The crate's core premise is demonstrably real. **Recommendation for the spike report: GO.**

**Files:**
- `src/backend.rs` — the three bridge impls for `Uuid` against `MultiBackend`.
- `src/lib.rs` — `pub mod backend;`.
- `tests/bridge.rs` — `exercise_one_arm(&mut DualConnection, …)` (backend-agnostic) + `one_arm_on_sqlite` / `one_arm_on_postgres`.
- `Cargo.toml` — added diesel feature `returning_clauses_for_sqlite_3_35` (see finding 2).

**Key findings (primary spike deliverables — also in [[DDB-T-0004]]):**

1. **The bridge is trivial, not high-risk.** `cargo expand` shows the derive generates two *public* helpers that do the per-arm dispatch: `MultiBackend::lookup_sql_type::<ST>(lookup)` (for `HasSqlType`) and `MultiRawValue::from_sql::<T, ST>()` (for `FromSql`). Our three impls are one-liners over them — *identical in shape* to the impls the derive emits for the core types (Integer/Text/Bool/…). The feared `FromSql<_, MultiBackend>` RawValue-matching is already solved inside the derive; we never touch `MultiRawValue` variants directly. `crate::MultiBackend` and `crate::MultiRawValue` are re-exported (`pub use`), and `MultiBackend` is defined in-crate, so the impls are local — **no orphan problem**, as predicted.

2. **`returning_clauses_for_sqlite_3_35` is REQUIRED** (newly discovered). Without it the first `get_result` failed: `ReturningClause … QueryFragment<Sqlite, DoesNotSupportReturningClause>` not satisfied. `QueryFragment<MultiBackend>` for a RETURNING clause needs *both* arms to support RETURNING; diesel gates SQLite's behind this feature (needs SQLite ≥ 3.35; we have 3.53). **This must be a hard, non-optional diesel feature for the crate** — without it the whole one-arm-RETURNING premise fails on SQLite. Record in M0 feature-flag task and the eventual published crate's required features.

3. **Bool is already supported on `MultiBackend`** (answers a [[DDB-T-0004]] question early). The expand shows the derive's required core set includes `HasSqlType<Bool>` (also SmallInt/Integer/BigInt/Float/Double/Text/Binary/Date/Time/Timestamp). **`Uuid` is absent — it is exactly the gap.** ⇒ **drop `types::Bool` from M0 scope.**

4. **Native-shim first read:** bridging diesel's *native* `diesel::sql_types::Uuid` onto `MultiBackend` looks mechanically identical (same three one-liner impls), since the helpers are generic over `ST`. The open risk is import shadowing (`diesel::sql_types::Uuid` vs our `sql_types::Uuid`) and choosing SQLite's representation for the native type — not a trait-expressibility problem. Feasible; defer per vision constraint 4.

**Handoff to remaining M1 work (DDB-I-0003):** every later portable type is the same three-line bridge over `lookup_sql_type` + `from_sql`. Strongly mechanical → a `dualdb-macros` codegen pass is attractive but optional. `tests/bridge.rs` is the template for the ORM-level per-type test.
