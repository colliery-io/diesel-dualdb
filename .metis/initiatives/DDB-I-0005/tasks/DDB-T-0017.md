# Real-schema coverage: constraints, FKs, indexes, ALTER, escape hatch

## Parent Initiative

[[DDB-I-0005]]

## Objective

Make the generator handle the DDL real schemas actually use — not just bare `CREATE TABLE`. Constraints, relations, indexes, schema evolution (`ALTER`), and an escape hatch for genuinely per-backend DDL. Without this the MVP only handles toy schemas.

## Acceptance Criteria

## Acceptance Criteria

- [x] **Constraints:** `NOT NULL`, single + **composite** `PRIMARY KEY` → correct PG/SQLite DDL and `table! (id, author) { … }`. `DEFAULT`/`UNIQUE`/`CHECK` flow through the DDL **for free** via the AST-Display approach (preserved, untouched).
- [x] **Foreign keys:** column + table-level `REFERENCES` → emitted in both DDLs and `diesel::joinable!(child -> parent (fk_col))` + `diesel::allow_tables_to_appear_in_same_query!(...)` appended to `schema.rs`.
- [x] **Indexes:** `CREATE INDEX` carried verbatim into both migration trees (no `table!` impact).
- [x] **`ALTER TABLE`:** add/drop column applied to the model (`schema.rs` reflects final state — verified `nickname` shows up) and rendered per-backend with native type mapping on the added column; `ADD CONSTRAINT` FK updates the join graph.
- [x] **Per-backend escape hatch:** `-- dualdb:postgres` / `-- dualdb:sqlite` … `-- dualdb:end` blocks → raw DDL to that backend only (verified: a `USING gin` index reaches PG but **not** SQLite), ignored by the schema model. Unbalanced/nested/unknown directives are errors. Documented in the module rustdoc.
- [x] Tests on a realistic `users`/`posts` FK fixture (+ composite PK, index, escape block, ALTER) + error cases. `cargo test -p schema-gen` green (3 tests); whole workspace green; `angreal check all` clean.

**Architecture note:** DDL is rendered by **type-rewriting the parsed AST** (column types → `DataType::Custom(native)`) and re-emitting via `sqlparser`'s `Display`, so all non-type SQL (FKs, constraints, defaults, indexes, ALTERs) passes through unchanged — far more robust than re-rendering from a hand-built model. A separate model pass drives `schema.rs`. Expanded `map_type` with `SmallInt`/`Float`. Output DDL is single-line (sqlparser canonical form) — fine for migrations.

## Implementation Notes

### Technical Approach
Extend the model from [[DDB-T-0016]]: `Table` gains constraints/indexes/FKs; add an `Alter` application pass so the final `schema.rs` reflects post-ALTER state while each migration file still emits its own incremental DDL. `joinable!`/`allow_tables_to_appear_in_same_query!` derive from the FK graph. The escape hatch: parse directive comments (sqlparser will choke on truly non-standard DDL, so capture raw blocks by directive and pass them verbatim to the tagged backend, skipping the parser for those).

### Dependencies
[[DDB-T-0016]] (generator core). Largest of the three build tasks.

### Risk Considerations
Medium-high — this is where SQL's surface area bites. Keep a clear "unsupported → explicit error or escape-hatch" boundary rather than silently mis-generating. ALTER-applied-to-model is the subtle bit (incremental DDL vs final schema state). The escape hatch is the pressure valve for anything out of scope.

## Status Updates

### 2026-06-09 — Complete

Real-schema coverage in. The generator now handles FK relations, composite PKs, indexes, `ALTER`, and per-backend raw DDL.

**Key design pivot:** migration DDL is rendered by **type-rewriting the parsed AST and re-emitting via `sqlparser::Display`** (column types → `DataType::Custom(native_token)`), so everything that isn't a column type — FKs, `REFERENCES`, composite `PRIMARY KEY`, `DEFAULT`, `UNIQUE`, `CHECK`, `CREATE INDEX`, `ALTER` — is preserved verbatim. A separate `Model` pass (CreateTable + ALTER folding) drives `schema.rs` (markers, composite PK tuple, FK → joinable!/allow_tables). This covers far more SQL than re-rendering from a hand-built model would, with much less code.

**Escape hatch:** `-- dualdb:postgres` / `-- dualdb:sqlite` … `-- dualdb:end` blocks emit raw DDL to that backend only and are skipped by the schema model. Verified a `USING gin` index reaches PG but not SQLite.

**Verified end-to-end** on a users/posts fixture — PG (`UUID`/`JSONB`, gin index) vs SQLite (`BLOB`/`TEXT`, no gin), both with FK + composite PK; `schema.rs` has `posts (id, author)`, `joinable!(posts -> users (author))`, `allow_tables_to_appear_in_same_query!`. 3 unit tests + error cases green; `angreal check all` clean.

**Notes / boundaries:**
- DDL output is single-line (sqlparser canonical) — acceptable for machine-run migrations.
- Auto-derived `down.sql` only covers `DROP TABLE` for tables created in the migration; ALTER/index migrations should supply an explicit `down.sql` (the generator passes it through).
- `map_type` covers Uuid/Timestamp/Bytes/Json + Text/SmallInt/Integer/BigInt/Float/Bool; others error (or go via the escape hatch). Date/Time/Numeric/Decimal are future.

**Handoff to [[DDB-T-0018]]:** the generator is capable enough to express the crate's own test schemas (incl. an FK fixture) — dogfood it: real `migrations/` dir → generated `schema.rs` replacing the inline `table!`s, `angreal schema gen` task, README run story, CI freshness check.
