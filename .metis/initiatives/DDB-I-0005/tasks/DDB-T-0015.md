# Spike: logical-DDL → PG/SQLite DDL + schema.rs (proof of concept)

## Parent Initiative

[[DDB-I-0005]]

## Objective

Test direction B's feasibility cheaply: take a `CREATE TABLE` written in **logical DDL** and generate (a) Postgres native DDL, (b) SQLite native DDL, and (c) a diesel `table!` block typed with portable markers. Decide the **logical-DDL form** (real SQL parser vs keyword convention) and pin the **type-mapping table**. Outcome feeds the M2-vs-schema-gen sequencing call (does generating real schema reveal ergonomic needs?).

## Acceptance Criteria

## Acceptance Criteria

- [x] POC generator: workspace member `schema-gen` (bin) parses a logical `CREATE TABLE` and emits PG DDL, SQLite DDL, and a `table!` block.
- [x] **Form decided: `sqlparser` (real SQL AST).** Its `DataType` enum carries the logical types as first-class variants (`Uuid`, `JSON`/`JSONB`, `Bytea`, `Timestamp`, `Integer`, `BigInt`, `Boolean`, …) — compiled on the first try. No need for a keyword convention.
- [x] Mapping table (`map_type`) covers UUID/TIMESTAMP/JSON/BYTEA → portable markers, plus TEXT/INTEGER/BIGINT/BOOL passthrough to `diesel::sql_types::*`. Unknown types fall through to a panic (would become a clear error in the real tool).
- [x] Nullability (`NOT NULL` → bare marker, else `Nullable<…>`) and `PRIMARY KEY` detection both work and drive the `table! (pk) { … }` shape.
- [x] Sample (`uuid PK + text + json(nullable) + integer + timestamp`) produces correct PG (`UUID`/`JSONB`/`TIMESTAMPTZ`), SQLite (`BLOB`/`TEXT`), and a `table!` with `id -> Uuid`, `payload -> Nullable<Json>`, grouped `use` imports. Unit test (`cargo test -p schema-gen`) asserts all three. ✅
- [~] **Stretch (generated `table!` compiles via `#[dualdb::test]`):** not automated, but the generated block is **byte-for-byte the same shape** as the hand-written `table!`s in `tests/uuid.rs` etc. that already compile and round-trip — so compilation is established by structural identity. Full generate→write→compile loop is the real generator's job.
- [x] Spike report below.

## Implementation Notes

### Technical Approach
New workspace member `schema-gen` (bin) depending on `sqlparser`. Parse with `GenericDialect`; walk `Statement::CreateTable` columns; map each `DataType` via a static table to `(pg, sqlite, marker)`; render three strings. Keep it small and throwaway-quality — findings are the deliverable, not polish. `sqlparser` already has `DataType::{Uuid, JSON, Bytea, Timestamp, ...}` variants, so the AST should carry the logical types directly — that's the main thing to confirm.

### Dependencies
None blocking. Independent of M2. Uses the existing `diesel_dualdb::sql_types::*` markers as the target.

### Risk Considerations
Low (it's a spike). Watch: `sqlparser`'s `DataType` variant names/shapes vary by version (compile-driven); nullability default (SQL columns are nullable unless `NOT NULL`) must map to `Nullable<…>` in `table!`. If `sqlparser` proves awkward for logical types, the keyword-convention fallback is the finding, not a failure.

## Status Updates

### 2026-06-09 — Complete — ✅ GO on direction B

The spike works; `schema-gen/src/main.rs` turns one logical `CREATE TABLE` into PG DDL + SQLite DDL + a `table!` block. `cargo run -p schema-gen` and `cargo test -p schema-gen` both pass.

**Form decision: `sqlparser` (not a keyword convention).** The logical types are first-class `DataType` AST variants, so parsing is free and robust. The "real SQL parser vs convention" question is settled in favor of the parser.

**Mapping table proven** (logical → PG / SQLite / marker): `UUID`→`uuid`/`BLOB`/`Uuid`; `TIMESTAMP`→`timestamptz`/`TEXT`/`Timestamp`; `JSON`/`JSONB`→`jsonb`/`TEXT`/`Json`; `BYTEA`→`bytea`/`BLOB`/`Bytes`; `TEXT`/`INTEGER`/`BIGINT`/`BOOLEAN` passthrough to native + `diesel::sql_types::*`. Nullability and PK handled.

**Sample output (verbatim):**
```
# Postgres            # SQLite              # schema.rs
id UUID … PRIMARY KEY  id BLOB … PRIMARY KEY  id -> Uuid,
payload JSONB          payload TEXT           payload -> Nullable<Json>,
created_at TIMESTAMPTZ created_at TEXT        created_at -> Timestamp,
```
The `table!` groups `use` imports (`diesel::sql_types::{Integer, Text}` + `diesel_dualdb::sql_types::{Json, Timestamp, Uuid}`) — no name collisions because our markers and the natives we use have disjoint names.

### Answer to the sequencing question (the reason for the spike)

**Schema-gen does NOT materially inform M2.** The generated `table!` is *identical* to the schemas we already hand-write in the tests — so writing queries against a generated schema is the same experience as against a hand-written one. Generating real schema surfaced **no new query-ergonomic needs**. The only genuine adjacency is migration **running** (apply DDL on connect), which is separate from M2's query/connection helpers.

⇒ **The initial hunch is (cheaply) disconfirmed. Recommendation: do M2 next to finish v1; build the full generator post-v1.** It's now de-risked — GO on direction B, `sqlparser` is the tool, the mapping is proven.
