# Schema & migration codegen (MVP)

> **Status: ACTIVE — in the MVP, building now (ahead of M2).** Created 2026-06-08 from "do we have a way to build our `schema.rs`?". **Product decision (2026-06-09): this is part of the MVP** — "write once, run on both" isn't real for an adopter who can't generate their schema + migrations. So it's promoted from post-v1 into v1 and built **before** M2.
>
> **Approach — Direction B:** a single logical schema, expressed as **migration DDL**, generates both the per-backend migrations **and** `schema.rs`. Independent of the native-type shim (B emits portable markers).
>
> **Spike done ([[DDB-T-0015]]) — ✅ GO.** `sqlparser` is the chosen form (logical types are first-class AST variants); the type-mapping table is proven; one logical `CREATE TABLE` yields PG DDL + SQLite DDL + a portable-marker `table!`. (Aside: the spike also showed the generated `table!` is identical to our hand-written schemas, so this work doesn't change M2's design — the two are independent.) The `schema-gen` workspace crate is the seed to grow.

## Context

A `diesel-dualdb` adopter needs a Diesel `schema.rs` (`table!` definitions) for their app. The crate is a library — downstreams own their schema — but the portable-types model makes the usual generator path awkward, so a helper could be real value.

**Why `diesel print-schema` does not fit (verified against diesel CLI 2.3.6):**

1. **It reads one live database and emits that backend's *native* SQL types.** Run it against Postgres → a `uuid` column prints as `Uuid`; against SQLite → the same logical column (stored `BLOB`) prints as `Binary`. The two backends therefore produce **different** schema files. There is no single run that yields a unified schema.
2. **It cannot emit our portable markers.** print-schema has no per-column type-mapping. Its only customization knobs are `--patch-file` (a unified diff applied post-generation), `--import-types`, `--custom-type-derives`, and `--pg-domains-as-custom-types`. None maps "native `uuid`/`bytea`/`timestamptz`/`jsonb` (PG) **and** native `BLOB`/`TEXT` (SQLite) → one `diesel_dualdb::sql_types::*` marker". A `--patch-file` could rewrite types but would be hand-maintained and **per backend** (different native types to rewrite on each side).

**Current pattern (works, manual):** hand-write a single unified schema with `diesel::table!` typed by the portable markers — exactly what cloacina does (`crates/cloacina/src/database/schema.rs`, `id -> DbUuid`, `created_at -> DbTimestamp`) and what our own tests do inline. No CLI, no migrations.

**Key coupling — the native-type shim (vision open question 1 / risk 1):** if we bridge diesel's **native** `sql_types` (`diesel::sql_types::Uuid`, `Timestamptz`, `Jsonb`, `bytea`) onto `MultiBackend` (the "compatibility shim" the spike judged feasible but deferred), then a schema generated by `diesel print-schema` **against Postgres** — which uses those native types — would compile and run on `MultiBackend` directly, with the SQLite arm handled by the shim's per-backend encoding. That would let us reuse `diesel print-schema` (PG) verbatim, no portable markers, no patch file. **Schema-gen feasibility is therefore largely gated on the native-shim decision.** Resolve that first.

## Goals & Non-Goals

**Goals (to be firmed up at real planning time):**
- A reproducible way for an adopter to obtain a working unified `schema.rs` for both backends without hand-typing every column.
- Stay aligned with one source of truth (avoid the "three places a query can live" failure the vision rejects — same spirit for schema).

**Non-Goals (for the MVP build):**
- A bespoke migration *runner*. The generated per-backend trees are **standard diesel migrations**, so `diesel migration run` (or `diesel_migrations` embedded) applies them per backend — we generate, diesel runs. We just document the run story.
- A SQL feature superset. Cover the column types + the constraints/relations real apps need (PK, NOT NULL, unique, FK, indexes, ALTER); exotic DDL goes through the per-backend raw-DDL escape hatch.
- Schema *introspection* (reading a live DB). Source of truth is the logical migration files, full stop.

## Detailed Design

### Chosen: B — logical schema (as migration DDL) → migrations + schema.rs

**Single source of truth:** the migrations, written in a **logical DDL** — ordinary `CREATE TABLE`/`ALTER TABLE` SQL using *logical* column types (`UUID`, `TIMESTAMP`, `JSON`, `BYTEA`/`BLOB`→`BINARY`, …) rather than a backend's native types. From those, a generator produces:

1. **Per-backend migration trees** — concrete DDL with each backend's native types (PG: `uuid`/`timestamptz`/`jsonb`/`bytea`; SQLite: `BLOB`/`TEXT`/`INTEGER`), i.e. the cloacina-style separate trees, now **generated** instead of hand-maintained.
2. **`schema.rs`** — unified `diesel::table!` definitions typed with the `diesel_dualdb::sql_types::*` portable markers.

So one logical definition yields both the DDL you *run* and the Rust schema you *query against*, and they can't drift. This matches the crate's one-source-of-truth philosophy and is independent of the native-type shim (it emits portable markers, not native types).

**The work is a generator** that reads the logical migration DDL and emits the two outputs. Open design questions for the eventual spike:
- **Logical DDL form:** a constrained SQL subset parsed by a real SQL parser (e.g. `sqlparser`), vs. a lighter convention (recognized logical type keywords) over otherwise-passthrough SQL. Needs to parse at least `CREATE TABLE` + column types + PK/nullability; pass non-type DDL through with per-backend tweaks where needed.
- **Type mapping table:** logical type → (PG native, SQLite native, portable marker, nullability handling).
- **Packaging:** an `angreal` task? a small bin in the workspace? a `diesel-dualdb-cli`? (Lean: an angreal task first, promote to a bin if it earns it.)
- **Divergence escape:** how a migration expresses genuinely per-backend DDL (indexes, constraints) that can't be derived — pass-through blocks tagged by backend.

### Rejected / not chosen

**A. `diesel print-schema` (PG) + native-type shim.** Reuses diesel's generator and would also advance the "drop-in, no schema edits" goal, but it makes PG the schema source of truth, depends on the (still-unbuilt) native shim, and doesn't generate the SQLite migration DDL. Kept on record because the native-shim is a separate lever worth its own decision; not the path here.

**C. `print-schema` + maintained patch file.** Per-backend patches, brittle type rewrites (a real `Binary` column vs a `Uuid`-as-BLOB are indistinguishable). Rejected.

## Alternatives Considered

- **Do nothing / document the manual pattern only.** The honest v1 answer (hand-write the unified `table!`, cloacina-style). Always available as the floor; this initiative exists to do better *later*.
- **Vendor/extend `diesel_cli_ext` or a print-schema fork.** Out of appetite; revisit only if A/B both stall.

## Implementation Plan

Decomposed 2026-06-09 (spike [[DDB-T-0015]] done → GO). MVP tasks:

1. **[[DDB-T-0015]] Spike** — ✅ done. `sqlparser` + mapping table proven on one `CREATE TABLE`.
2. **[[DDB-T-0016]] Generator core** — grow `schema-gen` from one-string POC into a real tool: read a `migrations/` dir of logical DDL (multiple files/tables), emit per-backend migration trees (PG + SQLite, `up.sql`/`down.sql`) **and** a unified multi-table `schema.rs`. Mapping misses become clear errors, not panics.
3. **[[DDB-T-0017]] Real-schema coverage** — beyond `CREATE TABLE`: `NOT NULL`/unique/PK (incl. composite), foreign keys (+ diesel `joinable!`/`allow_tables_to_appear_in_same_query!`), indexes, `ALTER TABLE`; and the **per-backend raw-DDL escape hatch** for DDL that can't be derived.
4. **[[DDB-T-0018]] Packaging, dogfood & docs** — `angreal` task (e.g. `angreal schema gen`); dogfood by generating `schema.rs` for the crate's own test tables and replacing the inline `table!`s; README + the `diesel migration run` story; CI freshness check (generated output matches committed).

**Exit criteria:** from a single logical-DDL migrations dir, the tool generates per-backend migration trees **and** a unified `schema.rs` that compiles and round-trips on both backends; it's dogfooded by the crate's own tests; the run story is documented.
