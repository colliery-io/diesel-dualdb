# Returning / upsert helpers

## Parent Initiative

[[DDB-I-0004]]

## Objective

Smooth the write paths that span both backends: insert-`RETURNING` ergonomics and a **portable upsert** for the safe common subset of `ON CONFLICT`. Keep genuinely divergent upsert behavior in the explicit escape hatch.

## Acceptance Criteria

## Acceptance Criteria

- [x] **RETURNING baseline confirmed:** `insert_into(...).get_result()` already works on one arm (covered by `tests/uuid.rs::uuid_insert_returning`). No helper needed — it's already ergonomic.
- [~] **Portable upsert — NOT achievable on one arm (finding).** `on_conflict(...)` does **not** compile through `MultiBackend`: the `MultiConnection` derive sets `MultiBackend`'s on-conflict dialect to `DoesNotSupportOnConflictClause`, and (unlike `RETURNING`) there is **no feature to enable it** and no external way to impl the `SqlDialect` bit. So upsert is an **escape-hatch case**, not a unified-arm helper. (This is what DESIGN.md already listed as "upsert feature gaps".)
- [x] **Escape-hatch upsert proven:** `tests/upsert.rs` runs `ON CONFLICT DO UPDATE` + `DO NOTHING` via `match conn` on the concrete connections (query identical on both, kept DRY with a macro). Green on both backends.
- [x] **Documented:** `docs/how-to/diverge-per-backend.md` gains an "Upsert is one of these cases" section with the reason + the macro pattern.
- [x] `angreal check all` green.

**Net:** the task imagined a portable upsert helper; the honest outcome is a *negative finding* (diesel's MultiBackend can't do on-conflict) + the escape-hatch pattern + docs. No new abstraction — correct per principle 1. Upstreaming on-conflict support to diesel's `MultiConnection` is a possible future contribution (noted for the roadmap).

## Implementation Notes

### Technical Approach
Diesel exposes `.on_conflict(...).do_update()/.do_nothing()` for both backends; the portable upsert is likely a thin convenience over that (or just documentation + an example if diesel's API is already portable enough). Resist over-abstraction — the design (DESIGN.md §4, principle 1) says abstract only the safe common subset. This task may end up small (mostly docs + tests) if diesel's surface is already portable; that's a fine outcome to record.

### Dependencies
[[DDB-T-0020]] not required but natural to follow. Independent of escape/release tasks.

### Risk Considerations
Low. Main risk is scope creep into a leaky upsert abstraction — keep it to the common subset and lean on the escape hatch for the rest.

## Status Updates

### 2026-06-10 — Complete (with a finding)

Surveyed the write paths; the headline outcome is a **finding**, not a helper.

- **RETURNING** already works on one arm — no work needed.
- **Upsert does NOT work through `MultiBackend`.** Tried `insert().on_conflict().do_update()/.do_nothing()` through `DualConnection` → compile error: `DoesNotSupportOnConflictClause: PgLikeOnConflictClause` not satisfied; `ConflictTarget`/`DoUpdate`/`DoNothing` aren't valid `QueryFragment<MultiBackend>`. The derive hard-sets MultiBackend's on-conflict dialect to unsupported and there's no feature (cf. `returning_clauses_for_sqlite_3_35`) to flip it, nor an external way to impl the `SqlDialect` associated type. So upsert is genuinely escape-hatch.

**Delivered:** `tests/upsert.rs` proving the `match conn` upsert works on both backends (DO UPDATE + DO NOTHING, DRY via a macro), and a docs section. No new abstraction — resisting a leaky one is the right call (principle 1).

**Roadmap note:** enabling on-conflict on `MultiBackend` would be an upstream diesel `MultiConnection` contribution; worth filing.

**Handoff:** [[DDB-T-0022]] (escape helpers) is next and directly related — a tidy `dispatch` wrapper would make this exact upsert `match` read cleaner.
