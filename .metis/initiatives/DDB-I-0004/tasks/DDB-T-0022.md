# Escape-hatch helpers + when-to-diverge docs

## Parent Initiative

[[DDB-I-0004]]

## Objective

Make the rare two-arm `match conn { Pg(c) => …, Sqlite(c) => … }` path tidy and loud — a thin typed helper so divergence is readable and the compiler still forces both arms to be handled. Plus the canonical "when to diverge" guidance.

## Acceptance Criteria

## Acceptance Criteria

- [x] `DualConnection::dispatch(|pg| …, |sqlite| …)` (`src/escape.rs`) — an inherent method (no trait import) taking one closure per backend, both required, returning a value. Runs the active arm via `match`; doesn't hide which arm runs.
- [x] "When to diverge" guidance — `docs/how-to/diverge-per-backend.md` lists the genuinely-divergent cases (locking hints, CTE+DML RETURNING, FTS/JSON operators, **upsert**) and the rule that everything else stays one-arm.
- [x] Test on a real divergent case: `tests/upsert.rs::upsert_via_dispatch` uses `dispatch` for the upsert (which can't go through `MultiBackend`), green on both backends. Plus a `no_run` doc example on `dispatch`.
- [x] How-to cross-links the `dispatch` helper (added a "A tidier helper" section).
- [x] `angreal check all` green.

## Implementation Notes

### Technical Approach
Keep it minimal — the value is readability + exhaustiveness, not abstraction. A free function or an extension trait on `&mut DualConnection` taking two closures (one per concrete connection) is enough; it just matches internally. Don't let it become a place to hide divergence (principle 1: the escape must stay loud).

### Dependencies
Independent. Pairs with the existing how-to doc.

### Risk Considerations
Low. The only trap is building something that obscures the divergence it's meant to make visible — resist that.
