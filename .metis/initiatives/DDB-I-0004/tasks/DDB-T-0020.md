# Pool::connect + URL/scheme detection

## Parent Initiative

[[DDB-I-0004]]

## Objective

`dualdb::Pool::connect(&url)` — one call that detects the backend from the URL and returns an r2d2 pool yielding `DualConnection`. Fixes the spike finding that the derived `Connection::establish` naively tries Postgres-then-SQLite (which can misroute or be slow to fail on an ambiguous URL).

## Acceptance Criteria

## Acceptance Criteria

- [x] Pure `detect_backend(url) -> Option<Backend>`: `postgres://`/`postgresql://` → Pg; `sqlite://`/`file:`/`:memory:`/bare-path → Sqlite; other scheme (`mysql://`…) → `None`. 3 unit tests cover the forms.
- [x] `dualdb::Pool` (`src/pool.rs`): a custom `DualConnectionManager` (`r2d2::ManageConnection`) establishes the **detected** backend explicitly — no Pg-then-Sqlite fallthrough. `Pool::connect(url)` → pool; `pool.get()` → `PooledConnection` (derefs to `DualConnection`).
- [x] Builder passthrough: `Pool::builder().max_size(n).min_idle(..).connect(url)`; `Pool::inner()` exposes the raw r2d2 pool, and `DualConnectionManager` is public for full r2d2 control.
- [x] Integration tests (`tests/pool.rs`): `Pool::connect` round-trips on SQLite (`:memory:`, max_size 1) and Postgres (env-gated, ran on the container).
- [x] `angreal test all` + `angreal check all` green; rustdoc example on `Pool::connect`.
- [x] Typed error: `pool::Error::{UnknownUrl, Pool}` (Display + `std::error::Error`).

## Implementation Notes

### Technical Approach
Detection is a pure string function (parse scheme; treat anything filesystem-ish or `:memory:` as SQLite). For the manager: diesel's `ConnectionManager<DualConnection>` calls `DualConnection::establish`, which is the naive path — so to force the right arm, either (a) wrap a custom `r2d2::ManageConnection` that builds `DualConnection::Pg(PgConnection::establish)` / `::Sqlite(SqliteConnection::establish)` based on the detected backend, or (b) accept the derived manager but document the routing. Prefer (a) for correctness (it's the whole point of this task). Pool is `r2d2::Pool<…>`; re-export the pooled-connection type.

### Dependencies
Spike (DDB-T-0004) proved r2d2 over `DualConnection` works. No blockers. First M2 task.

### Risk Considerations
Low-medium. SQLite `:memory:` pooling pitfall (per-connection DBs) — document + use a shared/file URL in tests. The custom `ManageConnection` is the main code; mirror diesel's `ConnectionManager` behavior (ping/is_broken via the derive's `R2D2Connection`).

## Status Updates

### 2026-06-10 — Complete

`dualdb::Pool::connect(url)` ships — one call, backend detected from the URL, no naive fallthrough.

**Files:** `src/pool.rs` (detection + `DualConnectionManager` + `Pool`/`Builder`/`Error`), `src/lib.rs` (`pub mod pool; pub use pool::Pool;`), `tests/pool.rs` (+2 `Pool::connect` tests; note updated).

**Design:** went with the custom `r2d2::ManageConnection` (option (a)) — it establishes exactly the detected arm, so a SQLite URL never pays the failed Postgres attempt. `is_valid`/`has_broken` delegate to the derive's `R2D2Connection` (`ping`/`is_broken`). `Pool` is a thin newtype over `r2d2::Pool<DualConnectionManager>`; `Builder` proxies `max_size`/`min_idle` and `Pool::inner()` + public `DualConnectionManager` cover the rest. Error types resolved cleanly through `diesel::r2d2::{ManageConnection, PoolError, Error}`.
